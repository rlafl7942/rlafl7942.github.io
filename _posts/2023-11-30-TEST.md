# 📝 면접 예상 질문

## 📌  데이터베이스 & 관계형 데이터베이스 기본 개념

### 📎 면접 예상 질문

- 파일시스템과 데이터베이스의 차이점에 대해서 설명해주세요.
    - 파일 시스템
    데이터를 파일로 관리하는 시스템으로 응용프로그램마다 필요한 데이터를 별도로 관리한다.
    - 데이터베이스
    여러 응용 프로그램들의 통합된 정보들을 저장하여 운영할 수 있는 공용 데이터들의 묶음이다.
    
    파일시스템의 경우 데이터를 수정해야할 때 프로그램 자체를 수정해야하지만, 데이터베이스의 경우, DBMS를 이용하여 변경이 쉽다. 또 파일시스템의 경우 응용 프로그램마다 독립된 파일을 가지고 있어 중복 저장이 불가피하고 이로 인한 데이터 값의 불일치가 발생할 수 있다. 하지만 데이터베이스는 데이터를 조직적으로 통합하여 중복을 없애고 자료를 구조화하여 중복이나 데이터 값의 불일치가 발생하지 않는다. 
    
- 데이터베이스의 특징에 대해 설명해주세요.
    - 실시간 접근성 : 사용자의 요청에 즉시 처리하여 응답하여야 함
    - 계속적인 변화 : 항상 데이터를 최신으로 유지해야 함
    - 동시공유 : 여러 명의 사용자나 응용 프로그램이 동시에 같은 내용의 데이터를 이용할 수 있어야 함
    - 내용에 따른 참조 : 데이터를 참조할 때 주소가 아닌 내용으로 참조
- DBMS는 뭘까요? 특징에 대해 설명해주세요.
    
    데이터베이스를 관리하고 응용 프로그램들이 데이터베이스를 공유하며 사용할 수 있는 환경을 제공해주는 소프트웨어이다. 
    
    - 데이터 무결성 : 부적절한 자료가 입력되어 동일한 내용에 대해 서로 다른 데이터가 저장되지 않게 함
    - 데이터 보안성 : 허가된 사용자들만 데이터베이스나 데이터베이스 내부에 접근 가능
    - 데이터 일관성 : 삽입, 삭제, 갱신, 생성 후에도 저장된 데이터가 모순이 없고 일정해야 함
    - 데이터 회복성 : 장애가 발생했을 때 특정 상태로 복구되어야 함
    - 데이터 효율성 : 응답시간, 저장공간 활용 등이 최적화 되어야 함
    
- 스키마가 뭘까요? 3단계 데이터베이스 구조에 대해 설명해주세요.
    
    스키마란 데이터베이스의 구조와 제약조건에 관해서 전반적인 명세를 기술한 것.
    
    (개체의 특성을 나타내는 속성, 속성들의 집합으로 이루어진 개체, 개체 사이의 관계에 대한 정의와 제약조건)
    
    - 외부 스키마 :
        - 개개인의 사용자가 보는 개인적 스키마
        - 사용자나 응용 프로그래머에 의해 정의됨
    - 개념 스키마 :
        - DB에 저장되는 데이터와 그들의 관계를 표현하는 스키마
        - 하나의 DBMS에는 하나의 개념 스키마만 존재
        - 제약 조건, 접근 권한, 보안 정책, 무결성 규정에 관한 명세 정의
    - 내부 스키마
        - DB의 물리적 구조 정의
        - 물리적 장치에서 데이터베이스가 실제로 저장되는 방법을 표현하는 스키마
        - 단 하나만 존재
        - 시스템 프로그래머나 시스템에 의해 설계됨
- 데이터 독립성에 대해서 설명해주세요.
    - 논리적 독립성 :
    개념 스키마가 변경되더라도 외부 스키마에는 아무 영향이 없는 것.
        
        전체 데이터베이스의 논리적인 구조가 변경되어도 사용자를 위한 외부 스키마는 변경할 필요가 없음
        
    - 물리적 독립성: 
    내부 스키마가 변경되더라도 개념 스키마에는 영향이 없는 것.
        
        데이터베이스의 저장 구조가 변경되어도 직접적으로 영향이 없는 데이터베이스의 논리적 구조는 영향을 받지 않음
        
    
- RDBMS(관계형 데이터베이스 관리시스템)는 뭘까요?
    
    관계형 데이터베이스 관리시스템이란 관계형 모델을 기반으로 하는 DBMS이다. 
    
    관계형 데이터베이스 : 데이터가 하나 이상의 열과 행의 테이블에 저장되어 서로 다른 데이터 구조가 어떻게 관련되어 있는지 쉽게 파악하고 이해할 수 있도록 사전 정의된 관계로 데이터를 구성하는 정보 모음
    
- 릴레이션 스키마와 릴레이션 인스턴스에 대해서 설명해주세요.
    - 릴레이션 스키마
    릴레이션이 어떻게 구성되는지, 어떠한 정보를 담는지 등에 대한 기본적인 구조를 정의한다.
    - 릴레이션 인스턴스
    정의된 스키마에 따라 테이블에 실제로 저장되는 데이터의 집합
- 릴레이션의 차수와 카니덜리티에 대해 설명해주세요.
    - 차수
    릴레이션 스키마의 속성 수
    - 카니덜리티
    릴레이션 인스턴스의 각각의 행의 개수, 즉 튜플의 수
- 키(Key)에 대해서 설명해주세요. (슈퍼키, 후보키, 기본키, 대리키, 외래키)
    - 슈퍼키 : 테이블에서 각 행을 유일하게 식별할 수 있는 하나 또는 그 이상의 속성들의 집합 (유일성 만족)
    - 후보키 : 테이블에서 각 행을 유일하게 식별할 수 있는 최소한의 속성들의 집합 ****(유일성, 최소성 만족) → 기본키가 될 수 있는 후보들
    - 기본키 : 후보키 중에 선택한 하나의 키, NULL 값이나 중복된 값을 가질 수 없음
    - 대리키 : 기본키 지정하고 남은 후보키
    - 외래키 : 다른 테이블, 릴레이션의 기본 키를 참조하는 속성 또는 속성들의 집합 (참조 테이블의 기본 키와 동일한 키 속성을 지님)
- 무결성 제약조건에 대해서 설명해주세요. (도메인 무결성, 개체 무결성, 참조 무결성)
    - 도메인 무결성 
    각 속성의 값은 무조건 도메인에 지정된 값만을 가져야 함.
    - 개체 무결성
    기본 키를 구성하는 모든 속성은 NULL이 아니어야 하며, 오직 하나의 값만 존재해야 함.
    - 참조 무결성
    자식 릴레이션의 외래 키 값은 참조된 부모 릴레이션의 기본 키 값과 같아야 하며, 자식 릴레이션의 값이 변경될 때 부모 릴레이션의 제약을 받음.
- 사용했던 데이터베이스에 대해서 설명해주세요. (오라클DB, MySQL, MariaDB, MongoDB 등)
    
    Mongo DB
    
    - 오픈소스 비관계형 데이터베이스 관리 시스템
    - 테이블과 행 대신 유연한 문서를 통해 다양한 데이터 형식 처리, 저장
    - 관계형 데이터베이스 관리 시스템(RDBMS) 필요 X
    - MongoDB의 문서는 JSON 형식으로 지정되어 다양한 유형의 데이터를 저장하고, 동적 스키마 설계를 활용
    
    ❗MySQL과 비교
    
    MySQL은 구조화된 쿼리 언어로 보관된 데이터에 액세스 → 테이블을 활용하여 데이터베이스를 생성하는 데 스키마를 사용하므로, 값을 검색하고 적절히 쿼리하는 기능을 제공
    
    - MySQL : 정적인 특성 → 데이터 무결성 및 격리가 필수적인 상황에서 유용
    - MongoDB : 형식이 덜 제한적, 성능 우수 → 가용성과 속도를 최우선으로 하는 상황에서 유용
    
    - (MySQL을 사용했다면) MySQL 엔진에 대해서 설명해주세요.
    - (MySQL을 사용했다면) InnoDB에 대해 설명해주세요.
    

## ✏️ 개념 정리

### 📌 데이터베이스 Database

1. 통합된 데이터
    
    중복을 최소화하며 데이터를 하나로 통합
    
2. 저장된 데이터
    
    컴퓨터 저장장치에 저장된 데이터
    
3. 운영 데이터
    
    조직의 목적을 위해 사용되는 데이터로 업무를 위한 검색을 목적으로 저장된 데이터
    
4. 공용 데이터
    
    공동으로 사용되는 데이터
    

### 📌 데이터 독립성

1. **논리적 독립성**
    
    개념 스키마가 변경 → 외부 스키마 영향 X
    
    전체 데이터베이스의 논리적인 구조 변경되어도 관련된 응용 인터페이스 정보만 적절히 수정해주면 직접 관련이 없는 사용자를 위한 외부 스키마는 변경할 필요 X
    
    ⇒ 사용자가 전체 데이터베이스의 논리적 구조가 변경되었다는 사실을 알 필요 X
    
2. **물리적 독립성**
    
    내부 스키마 변경 → 개념 스키마 영향 X
    
    데이터베이스의 저장 구조가 변경되어도 관련된 저장 인터페이스 정보만 적절히 수정해주면 직접적으로 관련이 없는 데이터베이스의 논리적 구조는 영향 X
    

### 📌 키 Key

키는 데이터베이스에서 조건에 만족하는 튜플을 찾거나 순서대로 정렬할 때 다른 튜플들과 구별할 수 있는 유일한 기준이 되는 속성

1. **슈퍼키**
    
    테이블의 각 행을 유일하게 식별할 수 있는 속성들의 집합 → 유일성 만족
    
2. **후보키**
    
    슈퍼키 중 최소성을 만족하는 속성들의 집합 → 유일성, 최소성 만족
    
3. **기본키**
    
    후보키 중의 선택된 하나의 키 → NULL or 중복된 값 X
    
4. **대리키**
    
    기본키를 지정하고 남은 후보키
    
5. **외래키**
    
    다른 테이블의 기본키를 참조하는 키
    
- 유일성 : 하나의 키 값으로 튜플을 유일하게 식별할 수 있는 성질
- 최소성 : 키를 구성하는 속성들 중 꼭 필요한 최소한의 속성들로만 키를 구성하는 성질

### 📌 스키마 Schema

데이터베이스 전체 또는 일부의 논리적인 구조를 표현하는 것으로 데이터베이스 내에서 데이터가 어떤 구조로 저장되는지를 나타냄.

1. **외부 스키마** : 사용자 관점의 스키마
2. **개념 스키마** : 사용자와 DB 관리자 관점의 스키마
    - 데이터베이스에 실제로 어떤 데이터가 저장되었으며, 데이터 간의 관계는 어떻게 되는지를 정의하는 스키마
    - 전체 관점으로 한 개만 존재
    - 접근 권한, 보안 및 무결성 등에 관한 정의 포함
3. **내부 스키마** : 저장 장치와 DB 설계자 및 개발자 관점의 스키마
    - 개념 스키마를 물리적 저장 장치에 구현하는 방법을 정의
    - 물리적 구조 및 내부 레코드의 물리적 순서 등 표현

### 📌 무결성 제약조건

1. 도메인 무결성
    
    특정 속성 값이 그 속성에 정의된 도메인에 속한 값이어야 한다는 규칙
    
    ex) 성별 속성의 도메인은 남/여
    
2. 개체 무결성
    
    기본키는 NULL 값이나 중복된 값을 가질 수 없다.
    
3. 참조 무결성
    
    관계 대응이 안되는 외래 키 값을 가질 수 없다는 규칙
    
    ex) 업무 관계의 ‘사번’ 속성에는 직원 관계의 ‘사번’ 속성에 없는 값은 입력할 수 없다.
